# Reptopetrol

A petrol station franchise management web application built with Spring Boot and Thymeleaf.

## Overview

Reptopetrol is a full-stack management system for a network of petrol stations. It covers the operational domain (stations, fuels, suppliers, contracts, transports, employees, inventories) as well as monitoring (CRUD audit history, event log, purchases). Access is secured with stateless JWT authentication.

## Tech Stack

| Layer | Technology |
|---|---|
| Backend | Spring Boot, Spring MVC, Spring Security |
| Persistence | PostgreSQL (JPA/Hibernate), MongoDB |
| Messaging | RabbitMQ (Spring AMQP) |
| Frontend | Thymeleaf, Bootstrap 4 |
| Auth | JWT (cookie-based, stateless) |
| Build | Maven |
| Infrastructure | Docker Compose |

## Domain Model

### Management (PostgreSQL)

| Entity | Description |
|---|---|
| **Station** | A physical petrol station with a name, address, and number of pumps |
| **Store** | A product store linked to a station |
| **Fuel** | A fuel type (GAS, DIESEL, LPG) available at stations |
| **FuelSupply** | The quantity and price of a specific fuel at a specific station (composite key: stationId + fuelId) |
| **Product** | A non-fuel product sold in a store |
| **Inventory** | The quantity and price of a specific product in a specific store (composite key: storeId + productId) |
| **Supplier** | A fuel supplier with a name, address, and home country |
| **Contract** | A time-bound agreement between a supplier and the network, covering one or more fuels |
| **Transport** | A delivery from a contract to one or more stations, handled by a transport company |
| **Employee** | A station employee (CASHIER or ATTENDANT) with a wage and hire date |

### Monitoring (MongoDB)

| Collection | Description |
|---|---|
| **CRUDHistory** | Audit log of create, update, and delete operations |
| **EventHistory** | General event log |
| **Purchase** | Records of fuel and item purchases |

## Authentication

Authentication is stateless and cookie-based:

1. The user logs in via `POST /api/auth/login` and receives a JWT stored in an `HttpOnly` cookie named `jwt`.
2. Every subsequent request includes that cookie automatically.
3. `JwtAuthenticationFilter` reads the cookie, validates the token against the database, and populates the Spring Security context.
4. Logout revokes the token and clears the cookie.

## Running Locally

### Prerequisites

- Java 17+
- Maven
- Docker

### Steps

```bash
# 1. Start infrastructure (PostgreSQL, MongoDB, RabbitMQ)
docker compose -f docker_compose.yaml up -d

# 2. Run the application
./mvnw spring-boot:run
```

The application starts on `http://localhost:8080`.

## Management Pages

Each entity has three pages:

| Path | Description |
|---|---|
| `GET /api/{entity}/all` | List all records |
| `GET /api/{entity}/add` | Add form |
| `GET /api/{entity}/update/{id}` | Edit form (pre-filled) |

Form submissions are sent as JSON via `fetch` (managed by `management-form.js`) so that the Spring `@RequestBody` endpoints are satisfied. The delete action uses a standard HTML form with `method="POST"` and a hidden `_method=DELETE` field, processed by Spring's `HiddenHttpMethodFilter`.
